# Personal Knowledge Ops

这个 skill 用于把个人知识库维护做成一条连续流程，而不是一次只做一个小动作。

## 工作模式

每次调用时，先判断属于哪一类：

1. `bootstrap`
2. `full-sync`
3. `ingest-only`
4. `opportunity-scan`
5. `project-update`

如果用户没有明确限定范围：

- 若基础结构不存在，先按 `bootstrap`
- 若基础结构已存在，按 `full-sync`

## 顶层结构

一级目录固定为 5 个：

- `个人知识/`
- `个人技能/`
- `个人区域/`
- `外部领域内容/`
- `副业/`

### 1. `个人知识/`

这个目录用于沉淀个人知识文件、分类结果、摘要和知识索引。

建议至少包含：

- `README.md`
- `INDEX.md`
- 按主题拆分的知识文件或分类目录

### 2. `个人技能/`

这个目录用于沉淀技能卡、技能索引和可服务能力。

建议至少包含：

- `README.md`
- `INDEX.md`
- 各技能对应的技能卡

### 3. `个人区域/`

这个目录用于记录你的背景、约束、应用场景和个人汇总。

建议至少包含：

- `README.md`
- `INDEX.md`
- `个人汇总.md`
- 其他基础信息文件

其中：

- `个人区域/`：放年龄、性别、城市、岗位、行业/公司、身份约束、可服务区域等基础画像与场景信息
- `个人汇总.md`：汇总个人区域、个人知识、个人技能，作为后续外部搜索和副业判断的统一输入

### 4. `外部领域内容/`

这个目录用于沉淀外部信息。

外部信息收集前，必须先读取 `个人区域/个人汇总.md`，再基于其中内容决定搜索方向。

### 5. `副业/`

这个目录用于管理副业项目。

建议至少包含：

- `已确认可尝试/`
- `确认不做/`
- `建议的新项目/`

## 首次初始化

初始化时必须创建：

- `个人知识/`
- `个人技能/`
- `个人区域/`
- `个人区域/个人汇总.md`
- `外部领域内容/`
- `副业/`

并为目录创建必要的 `README.md` 或 `INDEX.md`。

## 知识文件处理规则

用户不需要把所有文件搬进知识库目录。

用户只需要提供：

- 本地文件路径
- 网盘链接
- 图片路径
- PDF / Word / Excel / PPT 路径

然后由 skill 自动完成：

1. 读取内容
2. 判断一级/二级知识分类
3. 生成摘要
4. 记录原始文件名
5. 记录原始路径或链接
6. 更新 `个人知识/` 下的知识目录

知识目录至少要沉淀这些字段：

- 一级分类
- 二级分类
- 摘要
- 对应文件名
- 原始路径或链接

如果内容暂时无法读取，也至少要先登记一条待提取记录。

## 技能处理规则

用户主动说“新增技能”或“更新技能”时：

1. 如果信息不足，先追问 3 到 5 个短问题
2. 整理技能卡草稿
3. 给用户确认
4. 确认后写入 `个人技能/`

## 个人区域处理规则

初始化后应采集并维护这些基础信息，并写入 `个人区域/`：

1. 年龄
2. 性别
3. 所在城市
4. 所处岗位
5. 所处行业或公司

这些信息：

- 只保存在本地文件夹中
- 用于后续副业判断和机会匹配
- 不会为了联网搜索而原样向外泄露用户个人信息

## 个人汇总规则

`个人区域/个人汇总.md` 是整个系统的关键中间层。

它必须汇总：

- 个人区域
- 个人知识
- 个人技能

并提炼出：

- 当前背景
- 当前知识资产
- 当前技能资产
- 当前可变现方向
- 当前限制与约束

以下任一内容变化时，都必须并行更新 `个人汇总.md`：

- `个人区域/`
- `个人知识/`
- `个人技能/`

## 外部搜索规则

外部领域内容的收集，不能直接泛搜。

必须先读取：

- `个人区域/个人汇总.md`

再决定：

- 搜什么方向
- 哪些方向和用户匹配
- 哪些方向适合进入副业候选

不要先泛搜热点，再勉强往用户身上套。

## 副业规则

副业模块应基于：

- 个人汇总
- 外部领域内容

来形成：

- 候选项目
- 不做项目
- 已验证项目

## 副业候选的二次验证

当基于个人汇总和外部领域内容匹配出某个副业候选后，不要立刻只凭主观判断把它归为“值得尝试”。

还应围绕这个副业候选，单独再做一轮联网验证。

验证目标是补充“它为什么值得试，或者为什么不值得试”的外部数据支持。

### 二次验证应优先搜索

- 是否有人已经在做类似方向
- 是否存在明确用户需求
- 是否有公开内容热度、搜索热度或讨论热度
- 是否有可见的变现方式或成交迹象
- 是否有明显竞争过强、门槛过高、转化过低的信号
- 是否有地域或行业上的现实约束

### 二次验证后的输出

对每个副业候选，尽量补上这些判断：

- 值得尝试的支持证据
- 不值得尝试的风险信号
- 更适合的切入角度
- 最小验证动作
- 当前结论：建议尝试 / 暂缓观察 / 不建议做

### 规则

- 第一次匹配解决“是否与你有关”
- 第二次联网验证解决“是否值得现在尝试”

不要把这两步混成一步。

## 首次初始化后的回复要求

初始化完成后，必须明确告诉用户：

- 根目录绝对路径
- `个人知识/`、`个人技能/`、`个人区域/`、`外部领域内容/`、`副业/` 的实际路径
- `个人区域/个人汇总.md` 的作用
- 知识文件以后只需要给路径或链接，不必搬文件
- 技能以后怎么新增或更新
- 个人区域信息以后怎么补
- 后续统一入口是 `@personal-knowledge-ops sync`

还应给出两组可复制命令：

### 真实示例

```text
@personal-knowledge-ops sync 请收录这个文件：C:\资料\某产品复盘.pdf
@personal-knowledge-ops sync 新增技能：游泳
@personal-knowledge-ops sync 请基于个人汇总扫描一遍外部机会，并更新副业建议
@personal-knowledge-ops sync 我这周测试了一个副业项目：发了3篇内容，有2个咨询，请同步项目进展并更新结论
```

### 可替换模板

```text
@personal-knowledge-ops sync 请收录这个文件：<文件路径或链接>
@personal-knowledge-ops sync 新增技能：<技能名称>
@personal-knowledge-ops sync 请基于个人汇总扫描一遍外部机会
@personal-knowledge-ops sync 我更新了一个副业项目的进展，请帮我同步
```

## 决策原则

- 优先增量更新
- 优先保留用户原始文件位置
- 文件不必集中搬运
- `个人区域/个人汇总.md` 是外部搜索的唯一统一输入
- 高时效内容必须联网验证

## 不要做的事

- 不要要求用户把所有原文件都搬进一个目录
- 不要只给建议，不落实到文件更新
- 不要把热点直接等同于机会
- 不要伪造用户没有给出的实验结果

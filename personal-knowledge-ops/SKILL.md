# Personal Knowledge Ops

这个 skill 用于把个人资料维护做成一条连续流程，而不是一次只做一个小动作。

它同时负责首次初始化。如果用户还没有个人知识运营目录，要先自动搭好结构，再进入后续同步流程。

## 适用范围

当用户有以下任一意图时使用：

- 首次搭建个人知识运营系统
- 新增了知识文件，需要检查、归档、摘要、更新索引
- 新增了技能，需要生成或更新技能卡
- 希望扫描外部热点、需求、趋势或副业机会
- 希望把个人知识、技能和外部机会做匹配
- 希望同步副业实验进展、沉淀结论
- 希望“把这一套自动跑一遍”

## 工作模式

每次调用时，先判断属于哪一类：

1. `bootstrap`
   首次配置。创建基础目录、默认说明文件、汇总文件和副业分栏结构。

2. `full-sync`
   从头到尾跑一遍：检查新增资料、技能、项目反馈，再做外部机会扫描和副业候选更新。

3. `ingest-only`
   只处理新增知识文件或技能，不做外部扫描。

4. `opportunity-scan`
   只基于现有个人库做一次外部机会扫描和匹配。

5. `project-update`
   只处理副业实验进展和结论沉淀。

如果用户没有明确限定范围：

- 若基础结构不存在，先按 `bootstrap`
- 若基础结构已存在，按 `full-sync`

## 首次配置结构

首次配置时，必须创建一套适合个人长期维护的中文目录结构。默认根目录可以由用户指定；如果用户没指定，就在当前工作区下创建一个清晰命名的根目录。

必须包含这 5 个主目录：

- `个人知识/`
- `个人技能/`
- `个人区域/`
- `外部领域内容/`
- `副业/`

### 1. `个人知识/`

这个目录用于放“有文件支持、能被提炼成知识”的内容。

首次配置时至少创建：

- `README.md`
  说明这个目录是做什么的、适合放什么。
- `INDEX.md`
  作为总汇总，列出当前知识主题、子目录和每个主题的简述。

### 2. `个人技能/`

这个目录用于放“以能力表现为主”的技能卡。

首次配置时至少创建：

- `README.md`
- `INDEX.md`

`INDEX.md` 要汇总当前技能卡目录，并对每项技能给一句话概述。

### 3. `个人区域/`

这个目录用于放“地域上下文”，例如深圳、本地人群、线下机会、城市资源。地域不是知识大类，而是应用环境。

首次配置时至少创建：

- `README.md`
- `INDEX.md`

`INDEX.md` 要汇总已有地域项及其用途概述。

### 4. `外部领域内容/`

这个目录用于沉淀外部领域内容。它必须以日期为基准保存联网获取到的相关内容。

首次配置时至少创建：

- `README.md`
- `INDEX.md`

`INDEX.md` 用于汇总已有日期条目，并概述每次扫描的主题和发现。

后续联网扫描外部内容时，优先按日期新增条目，例如：

- `2026-06-16.md`
- `2026-06-23.md`

每个日期文件记录当次获取到的外部热点、需求、趋势、可关联机会和简要判断。

### 5. `副业/`

这个目录用于管理副业项目，不是放杂项，而是做明确分栏。

首次配置时，必须创建 3 个子目录：

- `已确认可尝试/`
- `确认不做/`
- `建议的新项目/`

并且每个子目录都至少有：

- `README.md`
- `INDEX.md`

含义如下：

- `已确认可尝试/`
  已经判断值得试、正在试或准备试的项目。
- `确认不做/`
  明确不做或已验证不适合的项目。
- `建议的新项目/`
  当前建议的新项目候选。

同时，`副业/` 根目录也应创建：

- `README.md`
- `INDEX.md`

总汇总这三个分栏中的项目概况。

## 默认目录

如果已有既定目录，则优先沿用。

如果采用旧版目录结构，也可以兼容，但新建时优先使用中文结构。

兼容场景下，如果用户明确要求用旧结构，可以继续使用：

`outputs/personal-knowledge-base/`

## 版本与更新

这个 skill 应支持“版本检查 + 更新提醒”，但不应擅自覆盖本地安装内容。

优先读取这些文件：

- `VERSION.md`
- `CHANGELOG.md`
- `REPO-INFO.md`

如果 `REPO-INFO.md` 中存在仓库安装链接，且当前任务允许联网：

- 可以检查远端仓库中的版本信息
- 将远端版本与本地 `VERSION.md` 对比
- 如果发现远端版本更新，要明确提醒用户“有新版本可安装”
- 提醒时给出建议安装方式，而不是自动覆盖本地文件

如果缺少远端仓库信息，就只报告当前本地版本，不假装完成了远端检查。

## 重点检查位置

优先检查这些位置：

- 用户这轮消息里明确提到的文件路径或主题
- `inbox/files/`
- `inbox/new-skills.md`
- `side-projects/experiment-feedback.md`
- `个人知识/`
- `个人技能/`
- `个人区域/`
- `外部领域内容/`
- `副业/`

## 执行顺序

按下面顺序执行，除非用户明确只要其中一段。

### 0. 首次配置检查

先检查是否存在上述 5 个主目录及其基础说明文件、汇总文件。

如果不存在：

- 创建目录
- 创建每个目录的 `README.md`
- 创建每个目录的 `INDEX.md`
- 创建 `副业/` 的 3 个子目录及其说明文件、汇总文件

首次配置完成后，再继续执行本轮同步任务。

### 1. 检查新增输入

优先检查：

- 用户这轮消息里明确提到的文件路径或主题
- `inbox/files/` 下的新文件
- `inbox/new-skills.md` 中新增技能
- `side-projects/experiment-feedback.md` 中新增实验反馈

如果没有新增内容，要明确说明“本轮没有发现新的输入项”，不要假装处理过。

### 2. 处理知识文件

对每个新增文件：

- 判断属于哪个知识领域
- 生成 3 到 5 句摘要
- 提炼“可复用点”
- 必要时补充适用人群、主题标签、地域标签
- 更新对应知识目录的 `INDEX.md`
- 如果知识目录下已有主题子目录，优先增量更新对应主题
- 如果还没有合适主题，可以新建主题条目并在 `INDEX.md` 中登记

如果文件内容无法直接读取，至少根据文件名、路径和用户描述先建立占位记录，并说明信息不完整。

### 3. 处理技能

对每个新增技能：

- 判断它更适合独立技能卡还是挂到现有技能卡中
- 用“我能做到什么、适用场景、证据或案例、后续待补材料”的方式更新技能卡
- 识别它与哪些知识领域有关
- 更新 `个人技能/INDEX.md`

不要把技能写成纯理论定义，要尽量体现“用户能做什么”。

### 4. 处理副业实验反馈

对每条新增实验反馈：

- 识别对应项目
- 更新实验记录中的结果、结论、是否继续
- 如果原本没有该项目，就新建一条实验记录
- 并根据结果判断它应留在 `已确认可尝试/`、转入 `确认不做/`，还是仅保留为观察项

不要把“试过了但没效果”当成失败无价值；它本身就是结论。

### 5. 联网扫描外部机会

仅在 `full-sync` 或 `opportunity-scan` 时执行。

必须使用联网搜索，优先找：

- 和产品经理相关的服务、内容、培训、咨询机会
- 和公共营养相关的内容、咨询、轻服务机会
- 和 NASM-CPT 或健身训练相关的内容、带练、计划设计机会
- 与用户已存在技能相匹配的热点需求
- 如果用户提到地域，如深圳，要纳入本地场景判断

输出时不要只列热点，要进行筛选和判断。

扫描结果要按日期写入 `外部领域内容/` 下的新日期文件，并同步更新 `INDEX.md`。

### 6. 做匹配

对每个外部机会，至少回答：

- 为什么和用户匹配
- 对应哪个知识模块
- 对应哪个技能模块
- 适合什么人群
- 最小验证动作是什么
- 是否建议进入副业候选

### 7. 更新外部内容和副业库

需要更新：

- `外部领域内容/` 下的日期文件
- `外部领域内容/INDEX.md`
- `副业/建议的新项目/INDEX.md`
- `副业/已确认可尝试/INDEX.md`
- `副业/确认不做/INDEX.md`
- `副业/INDEX.md`

更新时以增量为主，不要抹掉已有内容。

## 输出要求

每次执行后，给用户一个简明结果：

- 本次发现了哪些新增项
- 更新了哪些文件
- 是否做了外部机会扫描
- 新增了哪些副业候选
- 当前 skill 版本，以及是否发现可更新版本
- 是否有需要用户补充的地方

如果本轮内容不足以做某一步，也要明确说明缺口是什么。

## 推荐触发方式

统一入口优先使用：

- `@personal-knowledge-ops sync`

兼容这些表达：

- `@personal-knowledge-ops 跑一遍`
- `@personal-knowledge-ops 初始化我的个人知识运营系统`
- `帮我同步一下整个个人知识库`
- `我新增了几个文件，顺便看看有没有新的副业机会`
- `把这轮资料、技能和项目进展都更新一下`
- `根据我现在的库再扫一遍外部机会`

## 决策原则

- 优先增量更新，不做大重写
- 优先保留用户已有结构
- 地域信息默认作为标签或独立上下文目录，不把它混成知识主题
- 对副业机会先看匹配度和验证成本，再看想象空间
- 高时效内容必须联网验证，不凭记忆输出

## 当用户输入很少时

如果用户只说“跑一遍”或 `sync`，就按 `full-sync`：

1. 检查收件箱和反馈文件
2. 更新知识和技能
3. 联网扫描外部机会
4. 生成或更新副业候选
5. 返回本轮变更摘要

如果用户首次使用且只说“初始化”或“搭起来”，就按 `bootstrap`：

1. 创建 5 个主目录
2. 为每个目录创建说明文件和汇总文件
3. 为副业目录创建 3 个子目录及其说明文件、汇总文件
4. 返回初始化结果和后续使用方式

## 不要做的事

- 不要把所有内容都塞成一份长文
- 不要只给建议，不落实到文件更新
- 不要把热点直接等同于机会
- 不要伪造用户没有给出的实验结果
